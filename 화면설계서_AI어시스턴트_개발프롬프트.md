# 화면설계서 AI 어시스턴트 — 개발 요청 프롬프트

> 이 문서를 클로드 코드에 그대로 붙여넣어 사용하세요.

## 0. 배경 및 목적

나는 서비스기획(UX/PM) 취업 준비생이고, 인턴/포트폴리오 프로젝트에서 화면설계서(와이어프레임 스펙 문서)를 PPT로 작성해왔다. 화면설계서를 쓸 때마다 반복적으로 UI 패턴 고민을 한다. 예: FAQ를 아코디언으로 할지 별도 페이지로 할지, 필터/검색창을 어디에 배치할지, 폼 유효성 검사를 언제 보여줄지, 빈 상태/에러 상태를 어떻게 디자인할지 등등. 이런 고민의 경우의 수는 계속 늘어나기 때문에, 몇 개 패턴만 다루는 도구가 아니라 **새로운 고민이 생길 때마다 스스로 지식을 넓혀가는 도구**를 만들고 싶다.

또한 완성된 화면설계서(PPT)를 업로드하면 간격, 오타, 폰트, 컴포넌트 일관성 같은 걸 검토해주는 기능도 필요하다.

이 프로젝트 자체가 서비스기획자 포트폴리오 산출물이기도 하므로 코드 품질과 UI에도 신경 써줬으면 한다.

## 1. 아키텍처 원칙 — 서버 없는 단일 파일

**절대 조건: 서버 실행 없이 HTML 파일 하나(또는 몇 개의 정적 파일)를 더블클릭해서 바로 쓸 수 있어야 한다.** `npm run dev`, 백엔드 서버 기동 같은 절차가 필요 없어야 함.

- Claude API는 브라우저에서 `fetch()`로 직접 호출 (Anthropic API는 브라우저 CORS를 지원하는 헤더 옵션이 있으니 확인 후 적용, 안 되면 사용자가 자기 API 키를 입력하는 방식으로 처리)
- API 키는 화면 상단 설정 영역에 입력받아 `localStorage`에 저장 (개인용 도구이므로 이 정도 보안 수준으로 충분)
- PPT 파싱은 `.pptx`가 사실 zip 파일이라는 점을 이용해 브라우저에서 `JSZip`(CDN)으로 열고, 내부 XML(`slide1.xml` 등)을 파싱해서 도형 좌표(x/y/width/height), 텍스트, 폰트 크기/폰트명을 추출
- 슬라이드를 "이미지"로 렌더링해서 보는 Vision 분석(정렬감, 여백, 시각적 위계 판단)은 브라우저 JS만으로 PPT→이미지 변환이 어려우므로, **사용자가 파워포인트에서 슬라이드를 PNG로 내보내서 이미지로 업로드하는 방식으로 우회**한다. 이미지 업로드 시 Claude Vision(base64 인코딩)으로 분석.
- 데이터 저장(룰셋, 컴포넌트 라이브러리)은 전부 `localStorage` + JSON export/import 기능으로 처리. 백엔드 DB 없음.

## 2. MVP 범위

1. **패턴 컨설턴트**: 채팅으로 상황을 설명하면 UI 패턴을 추천. 룰셋 기반 + 부족하면 웹 검색 보강 + **새로 찾은 패턴은 룰셋에 저장해서 다음부터 재사용**
2. **QA 리뷰어**: PPT 파일 업로드(좌표/텍스트 분석) + PNG 이미지 업로드(선택, Vision 분석) → 간격/오타/폰트/UX 이슈 리포트

향후 확장(이번 스프린트 제외): 컴포넌트 일관성 체커(Figma 필요), 초안 자동 생성.

## 3. 패턴 컨설턴트 — 카테고리 체계와 자가확장 룰셋

### 3-1. 왜 하드코딩 목록이 아니라 체계가 필요한가

화면설계서 작성 중 마주치는 UI 패턴 고민은 FAQ/필터/검색 3개로 끝나지 않는다. 아래는 실제로 반복적으로 등장하는 **고민 카테고리**다. 각 카테고리 안의 구체적 사례는 무한히 늘어날 수 있으므로, 카테고리 체계 + 시드 규칙 + 자가확장 메커니즘으로 설계해야 한다.

| 카테고리 | 예시 고민 |
|---|---|
| 내비게이션/구조 | GNB/LNB 구성, 탭 vs 사이드 메뉴, 브레드크럼, 뒤로가기 동작 |
| 리스트/콘텐츠 표시 | 카드형 vs 리스트형 vs 테이블형, 정렬 옵션 위치 |
| 페이지네이션 | 페이지네이션 vs 무한스크롤 |
| 검색 | 검색창 위치, 자동완성, 최근 검색어, 검색 결과 없음 상태 |
| 필터 | 배치(사이드바/상단/바텀시트), 적용된 필터 표시 방식 |
| FAQ/도움말 | 아코디언 vs 별도 페이지 vs 탭 |
| 폼/입력 | 유효성 검사 타이밍(즉시 vs 제출 시), 필수 항목 표기, 단계형 vs 단일 폼, 에러 메시지 위치 |
| 모달/오버레이 | 모달 vs 바텀시트 vs 풀페이지 vs 토스트, 파괴적 액션 확인 패턴 |
| 상태 디자인 | 빈 상태(empty state), 에러 상태, 로딩 상태(스켈레톤 vs 스피너) |
| 버튼/CTA | 주/보조 버튼 위계, 고정(sticky) CTA vs 인라인 |
| 알림 | 토스트 vs 배너 vs 뱃지 |
| 온보딩 | 툴팁 투어 vs 모달 워크스루 vs 빈 상태 가이드 |
| 인증/가입 | 로그인/회원가입 패턴, 소셜 로그인 배치, 비밀번호 재설정 |
| 프로세스형 화면 | 스텝 인디케이터, 약관 동의 UI, 본인인증 단계 (금융/공공기관 서비스에 특히 관련 — 신청·심사 플로우) |
| 예외처리 | 네트워크 오류, 권한 없음, 데이터 없음 등 엣지케이스 화면 |

**구현 지침**: 초기 빌드 시 각 카테고리마다 2~3개씩 시드 규칙을 웹 검색(Baymard Institute, Nielsen Norman Group, Material Design 3, Apple HIG 등 신뢰도 높은 소스 우선)으로 조사해서 채워줘. 아래 스키마를 따르되, 카테고리는 위 표에 없는 것도 자유롭게 추가 가능하도록 설계할 것.

```json
{
  "id": "faq-accordion-01",
  "category": "FAQ/도움말",
  "condition": "항목 10개 미만, 단일 카테고리",
  "recommendation": "아코디언",
  "source": "Nielsen Norman Group",
  "rationale": "점진적 정보 노출로 인지 부하 감소",
  "avoid_when": ["대부분의 콘텐츠에 접근이 필요한 경우", "콘텐츠가 다단계로 복잡한 경우"],
  "implementation_notes": ["헤딩과 아이콘 모두 클릭 가능해야 함", "아코디언 중첩 금지"],
  "created_by": "seed"
}
```

### 3-2. 자가확장 메커니즘 (핵심 기능)

1. 사용자가 채팅으로 질문 → 로컬 룰셋에서 카테고리/조건 매칭 시도
2. 매칭되는 규칙이 없거나 불충분하면 → Claude API의 web_search로 실시간 조사
3. 조사 결과로 답변한 뒤, **화면에 "이 답변을 룰셋에 저장할까요?" 버튼을 띄워서 저장 여부를 확인** (자동 저장하지 말고 사용자 확인 거칠 것 — 잘못된 정보가 쌓이는 것 방지)
4. 저장 시 위 스키마 형식으로 `created_by: "user-session"`을 붙여 `localStorage`에 추가
5. 룰셋은 언제든 JSON으로 export/import 가능하게 해서, 다혜님이 직접 검토·수정하거나 백업할 수 있도록 함
6. 설정 화면에 룰셋 전체를 카테고리별로 보고 편집/삭제할 수 있는 간단한 뷰어 제공

## 4. QA 리뷰어

**입력**: `.pptx` 파일 (필수) + 슬라이드 PNG 이미지 (선택, Vision 분석용)

**처리**:
1. JSZip으로 pptx 열어서 슬라이드별 XML 파싱 → 도형 좌표, 텍스트, 폰트 크기/종류 추출
2. 좌표 기반 규칙 검사: 정렬 오차(동일 라인 요소 y값 불일치), 간격 불균형(반복 요소 간 간격 편차), 폰트 크기/종류 불일치
3. 텍스트 맞춤법/오타 검토 (Claude API 활용)
4. PNG 이미지가 업로드된 경우, Claude Vision으로 여백/위계/정렬감 등 시각적 판단 추가

**출력**: 슬라이드별 리포트, 이슈를 심각도(오류/권장/제안)로 분류, 슬라이드 번호와 대략적 위치 표시

## 5. UI 요구사항

- 단일 화면, 좌측 채팅(패턴 컨설턴트) / 우측 탭(QA 리뷰어 업로드+리포트, 룰셋 뷰어, 설정)
- 상단에 API 키 입력 필드 (localStorage 저장, 최초 1회만 입력하면 유지)
- 심플한 디자인, 다크모드/반응형 불필요 (개인 로컬 사용 전제)
- 리포트는 심각도별 색상 구분 정도만 (과한 디자인 지양)

## 6. 구현 순서 제안

1. 단일 HTML/JS 골격 세팅 (CDN으로 JSZip, Anthropic API 호출용 fetch 로직)
2. QA 리뷰어: pptx 업로드 → 좌표/텍스트 추출 → 규칙 기반 리포트 (Vision은 이미지 업로드 옵션으로 후순위 추가)
3. 패턴 컨설턴트: 카테고리별 시드 룰셋 조사 및 내장 → 채팅 UI → 자가확장 저장 기능
4. 룰셋 뷰어/편집 화면
5. 두 기능 한 화면에 통합, README에 사용법(파일 여는 법, API 키 발급 안내 링크) 정리

## 7. 향후 확장 (이번 스프린트 제외)

- **컴포넌트 일관성 체커**: Figma REST API로 마스터 컴포넌트-인스턴스 속성 비교. PPT는 구조적으로 어려움.
- **초안 자동 생성**: PPT는 `python-pptx` 등으로 서버사이드 생성이 필요해질 수 있음(브라우저에서 pptx 생성 라이브러리로 시도 후 한계 있으면 그때 서버 고려). Figma는 REST API가 읽기 전용이라 별도 Figma 플러그인(Plugin API) 개발 필요 — 별도 프로젝트로 분리.

## 8. 참고 리서치 소스

- Baymard Institute — 이커머스 필터/검색 UX 대규모 벤치마킹 연구
- Nielsen Norman Group — 아코디언, FAQ, 폼, 상태 디자인 등 가이드라인
- Jakob's Law, Google Material Design 3, Apple Human Interface Guidelines — 일반 UI 관습의 근거

이 문서의 룰셋 예시는 시작점일 뿐이며, 실제 커버리지는 섹션 3-2의 자가확장 메커니즘을 통해 사용하면서 계속 넓어지는 것을 전제로 설계할 것.
